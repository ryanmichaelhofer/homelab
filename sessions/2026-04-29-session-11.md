## T-Deck Cypher M8K map tiles (Scout)

- Diagnosed SD card failure: GPT partition table not readable by MUI firmware -- must be MBR
- Diagnosed ExFAT cluster size problem: default 128KB clusters caused 38x write amplification (477MB source -> 18GB on disk)
- Fixed: zero partition header with dd, then newfs_exfat -c 32 (16KB clusters = 32 sectors x 512 bytes)
- Final SD: ExFAT MBR 16KB clusters, 8.1GB used of 119GB
- Four map styles deployed to /maps/<style>/z/x/y.png:
  - atlas: official z1-6 worldwide (meshtastic/device-ui)
  - usa: full US z1-12 (torrent hash 3e2b9c010949d35710be8bf62f838ecb5e07cff2, 1.55GB)
  - osm: Hampton Roads z2-16 (MOBAC SQLite + map-tile-downloader Standard OSM polygon)
  - topo: Hampton Roads z13-16 partial (OpenTopoMap rate-limited at ~10k of 59k tiles)
- Tooling: MOBAC 2.3.3, map-tile-downloader (Flask/mattdrum), Transmission, custom Python extraction script
- Key lessons:
  - MUI firmware requires MBR not GPT
  - ExFAT default cluster size (128KB) is catastrophic for tile workloads -- always use 16KB (newfs_exfat -c 32)
  - newfs_exfat -c flag takes sectors not bytes
  - newfs_exfat refuses to reformat existing exfat partitions -- must zero first 10MB of partition first
  - OpenTopoMap rate-limits aggressively; Standard OSM handles bulk downloads better
  - MUI map tiles not found error at high zoom -- zoom out to z6 first
- Gallbush-Command (LilyGO T-Deck Cypher M8K) confirmed working -- 4 styles render on device
